# Project Notes

> 한줄 요약: 청춘캠퍼스의 서비스 구조, 신청 데이터 흐름, 관리자 기능, 자동매칭 규칙, 보안 원칙을 최종 기준으로 정리하는 마스터 문서입니다.

이 문서는 `청춘캠퍼스` 프로젝트의 구현 기준 문서입니다.
기본 실행 환경은 `휴대폰 모바일 브라우저`입니다.

현재 서비스 방향은 하나의 서비스 안에서 아래 두 흐름을 분리해 운영하는 구조로 확정합니다.
- `로테이션 소개팅`
- `1대1 소개팅`

## 1. 프로젝트 개요
- 프로젝트명: 청춘캠퍼스
- 한 줄 설명: 캠퍼스를 청춘으로 채우자
- 대상 사용자: 대학생
- 기본 접속 환경: 휴대폰 모바일 인터넷 브라우저
- 핵심 사용자 행동:
  - 메인 페이지에서 원하는 서비스 선택
  - 로테이션 소개팅 신청
  - 1대1 소개팅 신청
  - 후기 작성 및 후기 확인

## 2. 서비스 분리 운영 방향

### 2-1. 로테이션 소개팅
- 성격: 행사 신청형 서비스
- 운영 방식:
  - 남자 11명, 여자 11명 모집
  - 2~3시간 동안 단체 소개팅 진행
  - 회차별로 신청 관리
- 핵심 로직:
  - 정원 관리
  - 성별별 마감 관리
  - 대기열 처리
  - 관리자 페이지에서 신청 목록과 확정 상태 관리

### 2-2. 1대1 소개팅
- 성격: 프로필 수집 + 매칭형 서비스
- 운영 방식:
  - 지원자의 기본 정보와 선호 조건을 입력받음
  - 선호 스타일을 우선순위 형태로 입력받음
  - 저장된 데이터를 기반으로 자동매칭 후보 생성
- 핵심 로직:
  - 프로필 데이터 수집
  - 하드 조건 필터링
  - 선호 스타일 우선순위 점수화
  - 관리자 검토 기반 최종 확정
  - 필요 시 수동매칭 지원

## 3. 확정 아키텍처
- 프론트엔드: React 기반 화면 구성 유지
- 배포 형태: 정적 웹 배포
- 백엔드: Supabase 기준으로 확정
- 데이터베이스: Supabase Postgres
- 기본 실행 환경:
  - 모바일 브라우저 우선
  - PC는 보조 환경으로 지원
- 관리자 페이지:
  - `admin.html`을 별도의 비공개 페이지로 유지
  - 메인 버튼과 공용 상단 메뉴에서는 관리자 진입을 노출하지 않음
  - 공용 페이지에는 관리자 영역을 섞지 않음
- 로드 정책:
  - `index.html`, `rotation.html`, `rotation-apply.html`, `match.html`, `match-apply.html`, `reviews.html`, `guide.html`에서는 관리자 스크립트와 관리자 컴포넌트를 로드하지 않음
  - `admin.html`에서만 관리자 전용 스크립트와 관리자 컴포넌트를 로드
- 신청 데이터 흐름:
  - 신청 페이지에서 입력받은 데이터는 서비스 유형별로 저장
  - `admin.html`에서는 저장된 신청 데이터를 조회
  - 관리자 페이지는 `로테이션 관리`와 `1대1 매칭 관리`를 분리해서 제공
- 인증 정책:
  - 프론트엔드 코드에 관리자 비밀번호를 하드코딩하지 않음
  - 관리자 인증은 `커스텀 서버 세션` 또는 `Supabase Auth` 중 하나로 처리
  - 관리자 전용 조회/수정/매칭 실행은 서버 API 또는 Supabase Edge Functions를 통해 수행
- 구현 원칙:
  - 초기 버전은 파일 구조를 과하게 쪼개지 않음
  - 페이지 단위 + 기능 단위 정도로만 분리
  - 파일이 너무 커지거나 책임이 명확히 갈릴 때만 추가 분리

## 3-1. 상세 구현 요구 조건
- 모든 신청 데이터는 `서비스 유형`, `제출 시각`, `현재 상태`, `관리자 검토 여부`를 함께 추적할 수 있어야 합니다.
- 관리자 페이지 첫 화면에서는 `로테이션 신청 수`, `1대1 신청 수`, `자동매칭 후보 수`, `수동매칭 후보 수`를 요약 카드로 보여주는 구성을 우선합니다.
- 관리자 목록은 기본적으로 최신 신청순 정렬을 사용하고, 이름, 학교, 지역, 상태 기준 검색과 필터를 지원해야 합니다.
- 로테이션 신청은 회차별로 묶어서 보여주고, 확정/대기/마감 상태를 한눈에 볼 수 있어야 합니다.
- 1대1 신청은 `본인 스타일`과 `선호 스타일 우선순위`를 나란히 보여주어 자동매칭 근거를 즉시 판단할 수 있어야 합니다.
- 자동매칭은 `하드 조건 통과 -> 선호 스타일 우선순위 점수 계산 -> 보조 점수 반영 -> 후보 저장` 순서로 처리해야 합니다.
- 자동매칭 재실행 시 기존 후보를 덮어쓸지, 새 후보 이력으로 남길지 정책을 문서와 코드에서 일치시켜야 합니다.
- 수동매칭 시에는 `선택한 대상`, `수동매칭 사유`, `처리 관리자`, `처리 시각`을 반드시 기록해야 합니다.
- 관리자 액션은 `상태 변경`, `자동매칭 실행`, `수동매칭 생성`, `후보 승인`, `매칭 완료`, `보류 처리` 단위로 로그를 남기는 구조를 우선합니다.

## 8. 모바일 폼 UX 기준
- 기본은 모바일 우선 다단계 폼
- 한 페이지에 모든 항목을 넣지 않음
- 권장 단계:
  - 1단계: 기본 정보
  - 2단계: 성향/스타일
  - 3단계: 선호 조건
  - 4단계: 선호 스타일 우선순위
  - 5단계: 일정/동의
- 터치 기준 버튼 크기와 간격 확보
- 키패드 타입에 맞는 입력 필드 사용

## 8-1. 신청 폼 상세 요구 조건
- 필수 입력값이 누락되면 현재 단계에서 즉시 오류를 보여주고 다음 단계로 진행되지 않아야 합니다.
- 연락처는 숫자 입력 기준으로 검증하고, 잘못된 형식은 저장 전 차단해야 합니다.
- 선호 스타일 우선순위는 중복 선택을 허용하지 않아야 하며, 비어 있는 순위가 있으면 자동매칭 가중치 계산 시 명확한 기본값을 사용해야 합니다.
- 동일 사용자의 활성 신청이 이미 있는 경우 중복 신청 방지 정책을 적용해야 합니다.
- 제출 완료 후 사용자는 접수 완료 메시지와 현재 진행 단계 안내를 확인할 수 있어야 합니다.

## 10. 1대1 소개팅 매칭 구조
- 하드 조건 필터:
  - 나이 범위
  - 지역
  - 흡연/음주
  - 종교
- 선호 스타일 우선순위 점수:
  - 1순위 일치 시 가장 높은 점수 반영
  - 2순위 일치 시 중간 점수 반영
  - 3순위 일치 시 보조 점수 반영
- 보조 점수화:
  - 키 선호
  - 자기소개 키워드
  - 일정 적합도
- 결과 저장:
  - `matches` 테이블에 저장
- 운영 원칙:
  - 자동매칭은 선호 스타일 우선순위를 가장 먼저 반영
  - 관리자는 후보 결과를 검토하고 수동매칭으로 조정 가능

### 10-1. 관리자 페이지에서 필요한 매칭 기능
- `1대1 매칭 관리` 신청 목록 조회
- 신청자 상세 정보 확인
- 선호 스타일 우선순위 확인
- 자동매칭 실행 버튼 제공
- 자동매칭 결과의 점수, 사유, 후보 목록 확인
- 관리자 직접 선택 기반 수동매칭 실행
- 매칭 완료 처리 및 운영 메모 기록
- 위 기능은 모두 인증된 관리자 세션에서만 실행

### 10-2. 1대1 신청 상태 분류 권장
- `submitted`
  - 사용자가 신청서를 제출한 상태
- `reviewing`
  - 관리자가 신청 내용을 검토 중인 상태
- `waiting_match`
  - 검토가 끝났고 매칭 대기 중인 상태
- `candidate_found`
  - 자동매칭 또는 수동검토로 후보가 생성된 상태
- `matched`
  - 최종 매칭이 완료된 상태
- `hold`
  - 보류 또는 추가 확인이 필요한 상태
- `closed`
  - 취소, 만료, 운영 종료 등으로 더 이상 진행하지 않는 상태

### 10-3. 매칭 결과 상태 분류 권장
- `candidate`
  - 자동매칭이 생성한 후보 상태
- `manual_candidate`
  - 관리자가 수동으로 생성한 후보 상태
- `approved`
  - 관리자가 승인한 상태
- `completed`
  - 사용자 안내까지 포함해 최종 완료된 상태
- `cancelled`
  - 매칭 취소 또는 무효 처리된 상태

### 10-4. 자동매칭 및 수동매칭 상세 규칙
- 자동매칭 대상은 같은 성별, 자기 자신, 이미 최종 매칭 완료된 신청, 종료 상태 신청을 제외해야 합니다.
- 하드 조건이 하나라도 맞지 않으면 스타일 우선순위 점수 계산 전에 후보에서 제외해야 합니다.
- 스타일 우선순위 점수는 최소 `1순위 > 2순위 > 3순위` 순서를 보장해야 하며, 동점 시 보조 점수와 최근 신청 시각을 보조 기준으로 사용할 수 있습니다.
- 후보 저장 시 `총점`, `스타일 우선순위 점수`, `하드 조건 통과 여부`, `매칭 사유 요약`을 남겨야 합니다.
- 수동매칭은 자동매칭 후보가 없어도 실행 가능해야 하며, 자동 후보를 수동 후보로 승격하는 흐름도 허용할 수 있어야 합니다.
- 관리자 승인 전에는 사용자에게 상대 상세 정보를 노출하지 않아야 합니다.

## 11. Supabase 및 인증 확정안
- 사용 서비스:
  - Supabase Postgres
  - Supabase Auth 또는 커스텀 서버 인증
  - Supabase Storage는 이미지 업로드 시 사용 검토
  - Supabase Edge Functions는 관리자 전용 처리와 매칭 로직 처리 시 사용 검토
- 공용 페이지 저장 흐름:
  - 페이지에서 폼 입력
  - 프론트엔드 유효성 검사
  - Supabase API 또는 서버 API 호출
  - DB 저장
- 관리자 페이지 인증 흐름:
  - `admin.html` 진입
  - 서버 로그인 또는 Supabase Auth 로그인
  - 관리자 권한 검증 후 세션 발급
  - 인증된 상태에서만 신청 조회와 상태 변경 허용
- 보안 방향:
  - 모든 개인정보 테이블에 RLS 적용
  - 신청 데이터는 일반 사용자 조회 불가
  - 관리자만 신청 데이터 조회 가능
  - 후기 데이터는 승인된 데이터만 공개 조회 가능하도록 설계
  - 관리자 비밀번호와 시크릿 키를 프론트 코드에 넣지 않음
  - `service_role` 키는 서버 또는 Edge Functions에서만 사용
  - 공용 페이지에서 관리자 스크립트와 관리자 컴포넌트를 로드하지 않음

## 16. 운영/보안 요구사항
- 신청 데이터는 개인정보를 포함하므로 공개 조회 금지
- 관리자 외 신청 목록 열람 금지
- 관리자 페이지는 인증된 관리자만 접근 가능하도록 구성
- 프론트엔드에 관리자 비밀번호와 관리자 시크릿을 하드코딩하지 않음
- `service_role` 키는 클라이언트에 노출하지 않음
- 공용 페이지에서 관리자 스크립트와 관리자 컴포넌트를 로드하지 않음
- 관리자 페이지는 `로테이션 신청 관리`와 `1대1 매칭 관리`를 분리해야 함
- 로테이션 소개팅은 정원 관리가 핵심
- 1대1 소개팅은 선호 스타일 우선순위 기반 자동매칭과 수동매칭 지원이 핵심
- 자동매칭과 수동매칭 결과를 구분해 이력 저장
- 매칭 상태 변경 이력을 운영 메모 또는 로그로 남길 수 있어야 함
- 최종 매칭 완료 전에는 사용자에게 매칭 결과를 공개하지 않음
- 후기 데이터는 승인된 데이터만 공개 가능하도록 설계 권장
- 중복 신청 방지 방법 필요
- 스팸/봇 제출 방지 필요
- 관리자 세션 만료 시 재인증 없이 신청 데이터나 매칭 기능을 계속 사용할 수 없어야 함
- 관리자 전용 API는 서버 측 권한 검증 없이 호출되지 않도록 해야 함
