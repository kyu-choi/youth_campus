# 내부 기획 01: 프로젝트 개요

> 한줄 요약: 이 문서는 서비스의 방향, 핵심 범위, 관리자 분리 원칙, 이번 버전에서 반드시 구현할 기능을 정의합니다.

이 문서는 내부 기획용 문서입니다.
서비스의 큰 방향과 운영 구조를 정리합니다.

## 만들고 싶은 서비스
- 대학생 대상 소개팅 서비스
- 기본 사용 환경은 휴대폰 모바일 브라우저
- 하나의 서비스 안에서 두 흐름을 분리해 운영

## 서비스 흐름
- 로테이션 소개팅
  - 행사 신청형
  - 회차/정원 관리 중심
- 1대1 소개팅
  - 프로필 수집형
  - 선호 스타일 우선순위 기반 매칭 중심

## 관리자 운영 원칙
- `admin.html`은 별도의 관리자 전용 페이지로 유지
- 공용 페이지에는 관리자 UI를 섞지 않음
- 공용 페이지에서는 관리자 스크립트와 관리자 컴포넌트를 로드하지 않음
- 신청 페이지에서 받은 정보는 관리자 페이지에서 서비스별로 확인 가능해야 함
- 관리자 페이지는 `로테이션 신청 관리`와 `1대1 매칭 관리`를 분리해서 제공
- 1대1 소개팅은 선호 스타일 우선순위 기반 자동매칭과 수동매칭을 모두 지원
- 관리자 인증은 하드코딩 비밀번호 대신 `커스텀 서버 세션` 또는 `Supabase Auth`로 처리

## 현재 핵심 목표
- 로테이션 소개팅 페이지/신청 페이지 분리
- 1대1 소개팅 페이지/신청 페이지 분리
- 신청 데이터 저장과 관리자 페이지 연동
- `admin.html` 비공개 운영 페이지 유지
- 신청 상태 분류 및 매칭 관리 구조 정리
- 선호 스타일 우선순위 기반 자동매칭 구조 정리
- 수동매칭 기능 요구사항 정리

## 상세 요구 조건
- 이번 버전의 핵심은 `신청 -> 저장 -> 관리자 확인 -> 매칭 처리` 흐름이 실제로 이어지는 구조를 먼저 완성하는 것입니다.
- 로테이션 소개팅은 매칭보다 운영 관리가 중심이므로, 정원/대기열/회차 상태가 정확히 보이는지를 우선 기준으로 삼습니다.
- 1대1 소개팅은 `선호 스타일 우선순위`를 실제 매칭 판단에 반영해야 하며, 단순 문구 노출로 끝나면 안 됩니다.
- 관리자 페이지는 사용자용 화면과 완전히 분리된 운영 화면이어야 하며, 사용자 페이지에 운영용 문구나 버튼이 노출되면 안 됩니다.
- 이번 범위에서는 최소한 `신청 목록 확인`, `상세 조회`, `자동매칭`, `수동매칭`, `상태 변경`이 모두 연결되어야 합니다.
