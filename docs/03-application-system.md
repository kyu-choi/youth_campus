# 내부 기획 03: 신청 시스템

> 한줄 요약: 이 문서는 신청 폼 입력 항목, 검증 규칙, 저장 방식, 상태 관리, 관리자 페이지에서 확인해야 할 신청 데이터 기준을 정의합니다.

이 문서는 내부 기획용 문서입니다.
신청 시스템을 로테이션 소개팅과 1대1 소개팅으로 나누어 정리합니다.

## 공통 원칙
- 공통 프로필과 서비스별 신청 데이터를 분리
- 휴대폰에서 입력하기 쉬운 다단계 폼 구성
- 같은 폼으로 모든 신청을 처리하지 않음
- 공용 신청 페이지에는 관리자 영역을 섞지 않음
- 신청 페이지에서 받은 정보는 관리자 페이지에서 서비스 유형별로 확인
- 관리자 검토 기능은 `admin.html`에서만 제공

## 공통 프로필 정보
- 이름/닉네임
- 성별
- 나이
- 학교
- 학년
- 연락처
- 카카오톡 ID
- 지역
- 개인정보 동의

## 로테이션 소개팅 전용 정보
- 참가 회차
- 참가 가능 일정
- 참석 가능 여부
- 간단 소개

## 1대1 소개팅 전용 정보
- 키
- 본인 스타일
- 선호 나이 범위
- 선호 키 범위
- 선호 스타일 1순위
- 선호 스타일 2순위
- 선호 스타일 3순위
- 음주 여부
- 흡연 여부
- 종교 여부
- 자기소개

## 구현 시 주의할 점
- 로테이션 소개팅은 정원 관리가 핵심
- 1대1 소개팅은 매칭용 데이터 정합성과 검토 흐름이 핵심
- 자동매칭은 선호 스타일 우선순위를 가장 먼저 반영
- 관리자는 자동매칭 결과를 검토한 뒤 수동매칭으로 조정 가능
- 모바일에서 스크롤 피로가 크지 않도록 다단계 구성
- 관리자 비밀번호를 프론트 코드에 하드코딩하지 않음
- 관리자 접근은 서버 또는 Supabase 인증 세션을 전제로 설계

## 1대1 신청 상태 관리 권장
- `submitted`
  사용자가 신청서를 제출한 상태
- `reviewing`
  운영자가 신청 내용을 검토 중인 상태
- `waiting_match`
  검토가 끝났고 매칭 대기 중인 상태
- `candidate_found`
  자동매칭 또는 수동검토로 후보가 생성된 상태
- `matched`
  최종 매칭이 완료된 상태
- `hold`
  보류 또는 추가 확인이 필요한 상태
- `closed`
  취소, 만료, 운영 종료 상태

## 관리자 페이지에서 필요한 신청 관리 기능
- `로테이션 신청 관리`
  회차별 신청 목록, 확정, 대기열 관리
- `1대1 매칭 관리`
  신청 목록, 신청자 상세, 선호 스타일 우선순위 확인
- 자동매칭 실행 및 결과 확인
- 관리자 직접 선택 기반 수동매칭
- 매칭 완료 처리 및 운영 메모 기록
- 모든 관리자 기능은 인증된 세션에서만 실행

## 상세 요구 조건
- 모든 신청 폼은 필수값, 숫자 형식, 연락처 형식, 동의 체크 여부를 저장 전에 검증해야 합니다.
- 로테이션 신청은 선택한 회차가 마감 상태인지 확인하고, 정원 초과 시 대기열 상태로 저장할 수 있어야 합니다.
- 1대1 신청은 `본인 스타일`과 `선호 스타일 우선순위`를 분리 저장해야 하며, 같은 스타일을 여러 순위에 중복 선택할 수 없어야 합니다.
- 1대1 신청이 저장되면 관리자 페이지에서 `기본 정보`, `자기소개`, `본인 스타일`, `선호 스타일 우선순위`, `현재 상태`를 함께 볼 수 있어야 합니다.
- 상태값은 관리자 액션에 따라 명확히 바뀌어야 하며, 사용자가 제출했다고 바로 `matched`로 이동하면 안 됩니다.
- 신청 데이터가 저장되지 않았는데 관리자 화면에 임시 표시되는 구조는 허용하지 않으며, DB 저장 성공 후 관리자 목록에 반영되어야 합니다.

## 작업 순서
1. 공통 프로필 필드 확정
2. 로테이션 전용 필드 확정
3. 1대1 전용 필드 확정
4. 선호 스타일 우선순위 입력 구조 확정
5. 신청 페이지 UI 구성
6. 저장 로직 연결
7. 관리자 페이지에서 서비스별 신청 목록 연결
8. 1대1 신청 상태 관리 구조 연결
9. 선호 스타일 우선순위 기반 자동매칭 연결
10. 수동매칭 기능 연결
